    PR 批注建议

    概述

    文档和代码一样，需要持续改进，不断优化。作为审核人，在文档 PR 评审中发现的问题，无论优先级与否，都会进行批注。但作为文档的签入人员，则可以根据项目的工作安排，合理安排修改的优先级。

    方便大家可以通过简单的标识对于 PR 批注的优先级能够快速识别，特进行以下定义及明确相关的优先级。

    定义及优先级

    遗漏

    文档里面应该描述实际未描述的，即不符合文档完整性的情况。

    此类问题一般优先级高，PR 批准必须修改。

    批注时可以采用以下格式。

    遗漏：XXXX 参数遗漏。

    常见问题：

    • 信号、参数等接口遗漏
      • 遗漏某个参数/信号
      • 遗漏参数/信号中的某个项，比如默认值、类型、返回值等。
    • 参数约束或限制未描述
    • 信号的前置条件未描述
    • 异常处理或报警信息未描述
    • 未提供竞品信息
      • 未明确和竞品不同之处，或遗漏问题未放到 分析 章节
      • 未明确暂不实现及原因
      • 未明确不计划实现或删除及原因。（reference.md 分析章节）
    • 提供信息但不完整
      • 提示说报错但未提供报错信息
      • 明确了范围取值但未明确每个值的作用‘
      • PR 回复后未在文档里面更新。

    错误

    文档里面描述但和实际代码不符合的，即不符合文档正确/准确性的情况。

    此类问题一般优先级高，PR 批准必须修改。

    批注时可以采用以下格式。

    错误：XXXX 参数的限制描述错误，正确的是XXXX。

    常见问题：

    • 使用限制多余或不正确
    • 参数值错误
    • 概念定义模糊

    问题/建议

    文档里面描述，但模糊不清楚或有歧义的等等不符合本规范的情况。

    此类问题一般优先级中或低。

    建议：该语句不通顺，建议修改为 XXXX 。
提问：这个 XXX 表示的是？

    常见问题：

    • 多余的信息
    • 同类文档、章节、功能的组织、功能名等形式不统一
      • 功能名不一致
      • 文件名称和#标题名称不一致
      • 术语或概念不一致
      • 指令格式不一致
      • 各章节顺序不一致
      • 元数据格式不一致
      • 寻址定义有不一致性
      • 寻址里面同类型索引值不一致
      • 寻址类型不一致
    • 接口名称描述不简洁
    • 概述没有抓住重点
      • 太过于简单，信息量少。(要描述Why+What)
      • 比较啰嗦，句子不通顺。（referenc.md 对此处的要求比较具体。）
    • 中英文之间、中文与数字之间无空格
    • 强调 类型文本使用不一致
    • 标题层级不符合规范
    • 错别字
